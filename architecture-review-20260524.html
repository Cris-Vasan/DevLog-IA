1. Enum Constants — No Single Source of Truth

Strong
Files involved
server/src/services/tasks.js:3–5 server/src/routes/tasks.js:10–12 client/src/pages/ProjectView.jsx:7–24 server/src/db.js:19–21
Problem

Domain values — priority, category, status — are defined independently in the service, routes, frontend component, and DB schema. Four sources of truth. Adding "critical" priority means editing all four files and hoping the frontend gets a colour. The interface of this "enum knowledge" is as complex as its implementation (a list of strings) — a textbook shallow module.

Solution

Expose the valid values from the API at GET /api/meta/enums (or bake them into a shared constants.js on the server side and consume via the Vite proxy). The frontend fetches or imports the canonical list; colour maps live in one place keyed by value.

Benefits

Locality: one file to edit when a new category is added. Leverage: the API now self-documents its domain values; the client validates against them; tests can assert the enum surface hasn't changed.

+ route-level validation

const PRIORITY_COLORS = {...}

● db.js CHECK constraints
'low','medium','high' ← copy

2. DB Connection Hidden in Express App State

Strong
Files involved
server/src/index.js:10 server/src/routes/projects.js:15,24,29,35,41 server/src/routes/tasks.js:19,30,52,72 server/test/projects.test.js:13
Problem

Every route handler retrieves the database via req.app.get('db') — a string-keyed lookup on the Express app object. This is a runtime dependency hidden behind an opaque key. Tests must manually call app.set('db', db) before each suite. The seam is real (routes don't create their own db) but it's a bad seam: the interface is the entire Express app rather than a narrow db-shaped thing.

Solution

Use a router factory: each router module exports a function that accepts db and returns the configured router. app.js calls projectsRouter(db). The dependency is explicit, typed, and injected at wiring time — not fetched at request time.

Benefits

Locality: the db dependency is visible at the module's interface — no hidden app.get() calls. Leverage: tests pass a real in-memory db directly to the router, with no app.set() ceremony. Extending to a second db or logger follows the same pattern.

Before
// app.js
app.set('db', db)
app.use('/api', projectsRouter)

// routes/projects.js
router.get('/', (req, res) => {
  const db = req.app.get('db') ← hidden
  const projects = listProjects(db)
  ...
})
After
// app.js
app.use('/api', projectsRouter(db)) ← explicit

// routes/projects.js
module.exports = function(db) {
  const router = Router()
  router.get('/', (req, res) => {
    const projects = listProjects(db)
    ...
  })
  return router
}

3. Parent-Exists Check Duplicated in Every Child Route

Worth exploring
Files involved
server/src/routes/tasks.js:18–26 server/src/routes/tasks.js:29–32 server/src/routes/sessions.js (upcoming)
Problem

Both the GET and POST handlers for tasks call getProject(db, req.params.id) before doing any work. Sessions (issue 006) will repeat this pattern. Each route handler owns its own copy of "verify the project exists, return 404 if not" — three identical error responses, three identical getProject calls.

Solution

An Express middleware requireProject(db) that reads req.params.id, verifies the project, attaches it as req.project, and calls next() or returns 404. All child routes use it; none repeat the check.

Benefits

Locality: project-existence logic lives once. Leverage: sessions and future child routes get it for free. Tests for the middleware cover all routes at once.

Before — pattern repeats
router.get('/projects/:id/tasks', (req, res) => {
  const project = getProject(db, req.params.id) ← copy
  if (!project) return res.status(404).json(...)
  ...
})

router.post('/projects/:id/tasks', (req, res) => {
  const project = getProject(db, req.params.id) ← copy
  if (!project) return res.status(404).json(...)
  ...
})
After — middleware
function requireProject(db) {
  return (req, res, next) => {
    req.project = getProject(db, req.params.id)
    if (!req.project) return res.status(404).json(...)
    next()
  }
}

router.get('/projects/:id/tasks',
  requireProject(db), ← one place
  (req, res) => { /* use req.project */ }
)

4. Validation Split Between Route Layer and DB Constraints

Worth exploring
Files involved
server/src/routes/tasks.js:39–63 server/src/db.js:19–21 server/src/services/tasks.js
Problem

Enum validation lives in the route (returns HTTP 400) and in the DB schema (throws an exception). The service layer — the natural home for "is this a valid task?" — has no validation of its own. POST and PUT handlers both validate, but with slightly different logic. If the route validation is bypassed, a DB exception surfaces instead of a clean 400.

Solution

Move validation into the service functions as validate(fields) — returns { valid: true } or { error: '...' }. Routes call validate() and map the result to HTTP. The DB constraints remain as a safety net. Tests for validate() are pure functions: no HTTP, no db.

Benefits

Locality: validation logic is colocated with the domain model. Leverage: route handlers become thin — they validate, call a service, return JSON. Pure validation tests are fast and exhaustive.

Before
Route: POST /tasks
if (!title) → 400
if (!VALID_PRIORITIES.includes) → 400
if (!VALID_CATEGORIES.includes) → 400
→ createTask(db, ...)

Route: PUT /tasks/:id
if (priority && !VALID...) → 400 ← different!
if (status && !VALID...) → 400
→ updateTask(db, ...)

DB: CHECK constraint
→ throws exception if bypassed
After
services/tasks.js
validateCreate(fields) → {error?}
validateUpdate(fields) → {error?}

Route: POST /tasks
const err = validateCreate(req.body)
if (err) return res.status(400).json(err)
→ createTask(db, ...)

Route: PUT /tasks/:id
const err = validateUpdate(req.body)
if (err) return res.status(400).json(err)
→ updateTask(db, ...)

5. Services Untested in Isolation

Worth exploring
Files involved
server/src/services/projects.js server/src/services/tasks.js server/test/projects.test.js server/test/tasks.test.js
Problem

All 37 tests go through the HTTP stack via supertest. Service functions — listTasks(), updateProject(), createTask() — are never tested directly. This means: (a) tests are slower than needed, (b) a bug in listTasks()'s filter logic requires an HTTP call to uncover, (c) refactoring the route without touching the service can still break tests, making it hard to locate the regression.

Solution

Keep the integration tests (they're valuable). Add server/test/services/tasks.test.js that imports service functions directly and calls them with an in-memory db. These tests are pure, fast, and pinpoint failures to the service layer.

Benefits

Leverage: the service interface becomes the test surface — as it should be. The route tests shrink to verifying HTTP mapping (status codes, headers). Locality: a regression in listTasks filtering fails in exactly one place.

Before — all tests are HTTP
it('GET ?status= returns matching tasks', async () => {
  const res = await request(app)  ← HTTP roundtrip
    .get(`/api/projects/${id}/tasks?status=done`)
  expect(res.body).to.have.lengthOf(1)
})
After — service tests are direct
// test/services/tasks.test.js
it('listTasks filters by status', () => {
  const tasks = listTasks(db, pid, { status: 'done' })
  expect(tasks).to.have.lengthOf(1)  ← direct
})

// test/tasks.test.js (HTTP)
it('GET ?status= returns 200', async () => {
  const res = await request(app).get(...)
  expect(res.status).to.equal(200)  ← just HTTP
})

6. apiFetch() — Shallow Wrapper with Bad Ergonomics

Speculative
Files involved
client/src/api/client.js:3–16
Problem

The apiFetch() function spreads ...options after setting headers — any caller passing headers will silently override Content-Type. Error handling is generic (doesn't distinguish 400 from 500). The abstraction is thin enough that callers still need to know path strings, JSON serialisation, and the 204 special case.

Solution

Fix the header spread order now (1-line fix). Speculative: deepen the client by adding error classification ({ status, error, isValidation }) so hooks can distinguish "show field errors" from "show toast".

Benefits

Locality: one place for error shape decisions. Low priority — the app is single-user and errors are rare. The header bug is the concrete fix; the rest is speculative.

Top Recommendation

Start with #2 — Router Factory (DB Injection)

It's the smallest change with the widest impact. Right now every route handler is coupled to Express app state for its core dependency. The router factory pattern eliminates this seam and makes the tests simpler at the same time — no more app.set('db', db) boilerplate in every test suite. Sessions (issue 006) will add a third route module; fixing the pattern now costs one refactor instead of two.

After that: #1 (enum constants) before implementing filtering (issue 005) — adding filter controls in the UI will expose the frontend duplication immediately. #3 (requireProject middleware) naturally follows when sessions arrive. #4 (validation in services) and #5 (service unit tests) can accompany any new feature slice.

Candidates ranked by: leverage (how many callers benefit) × locality (how concentrated the change is) × timing (how soon the friction compounds).
